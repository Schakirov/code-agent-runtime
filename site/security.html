Security model

Honest framing: this is not a production-grade sandbox.

status: threat model documented; controls planned (Milestone 7)

A capable adversary, or an agent running with broad host permissions, can escape the controls described here. The goal is blast-radius reduction and observability for semi-trusted, deterministic agents on a developer machine.

Brief explanation

The security model protects developer secrets, files outside the workspace, the network, and the host from destructive commands — against bugs and over-eager agents, not against deliberately malicious native code.

Deeper technical explanation

In scope

  • An agent that reads or writes outside its workspace.
  • An agent that runs a dangerous shell command by mistake.
  • Accidental secret capture into traces or reports.

Out of scope

  • A deliberately malicious agent with native code execution and kernel exploits.
  • Side-channel, timing, or hardware attacks.
  • Multi-tenant isolation guarantees.

Current implementation status

Runtime containment is not implemented yet. Today's guards are repo-level .gitignore and .claude/settings.json deny rules plus the no-secrets-committed operating principle. Repo hygiene scanning lands in Milestone 1.

No security tests yet. Validation will assert blocked-vs-allowed behavior on unsafe demos.
