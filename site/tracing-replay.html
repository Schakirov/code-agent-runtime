Tracing & replay

A run that cannot be replayed from its trace is a bug, not a feature.

status: planned (Milestones 5–6) — not implemented yet

Brief explanation

Every run is recorded as a structured trace capturing what the agent did and what the runtime observed. Traces are the evidence behind every reported result.

Deeper technical explanation

A trace will include task metadata, workspace metadata, agent actions, tool calls, command stdout/stderr, file diffs, test results, timestamps, and exit status. Replay reconstructs the action timeline, tool calls, diffs, tests, and failure point — it does not need to re-execute every command to be useful.

Current implementation status

Not persisted yet, but the raw material exists. The runtime (Milestone 4) already returns a structured, JSON-serialisable RunResult: per-step records (the agent action, its arguments, whether it was allowed, and the structured tool result), the test result, the final diff, the score, and the outcome. That is the first draft of the trace schema. The dedicated trace schema and recorder that persist this to disk land in Milestone 5; inspection and replay tooling in Milestone 6.
