Task format

Versioned, declarative task definitions that pin everything a run needs.

status: implemented (Milestone 2)

Brief explanation

A task is a small, version-controlled file describing one unit of work: what to do, which fixture repo to do it in, what tools are allowed, under what limits, and how success is measured. Tasks are data only — loading one runs no agent and executes no command. The runtime that consumes them is built in later milestones.

Deeper technical explanation

The canonical format is JSON (*.task.json); YAML (*.task.yaml/.yml) is supported optionally and parsed only when PyYAML is installed, so the core package and default tests stay dependency-free. Fields:

FieldRequiredMeaning
versionyesSchema version (currently 1).
id, titleyesStable identity (lowercase, -_./ separators) and name.
promptyesInstruction given to the agent.
fixtureyesWorkspace snapshot path, relative to the task file.
scoring_methodyesOne of test_command, expected_files, diff_constraint, custom, none.
allowed_toolsnoSubset of the tool registry the agent may use.
timeout_secondsnoWall-clock budget (default 300).
test_commandcond.Required when scoring by command; shell string or argv list.
resource_limitsnocpu_seconds, memory_mb, max_output_bytes, network.
metadatanoFree-form tags (category, language, difficulty, ...).

Valid tasks load cleanly. Invalid tasks raise a single error that lists every problem (unknown/missing fields, bad types, unknown tools, unrecognised scoring method, unsupported version, bad limits) with the source file named — so an author fixes everything in one pass. allowed_tools and scoring_method are checked against closed vocabularies, turning typos into authoring-time errors.

Current implementation status

Implemented. The tasks/ package provides the schema, loader (JSON + optional YAML), and fixture resolution. Three example tasks ship with real fixtures, and the CLI can list and inspect them:

code-agent-runtime tasks list --dir tasks
code-agent-runtime tasks show bugfix/sum-range-off-by-one --dir tasks

Current test/result status

Covered by tests/test_tasks.py (CPU-only): schema validation for valid and invalid documents, JSON and optional-YAML loading, fixture resolution, discovery over the real tasks/ tree, and the tasks list/show CLI. The full suite passes with python3 -m pytest -q.

Open questions / limitations

  • How to express diff/scope constraints declaratively (the diff_constraint scoring method is named but not yet implemented; scoring lands in Milestone 8).
  • allowed_tools is validated against a hard-coded vocabulary that must stay in sync with the Milestone 3 tool registry.
  • Fixture paths are resolved relative to the task file; repo-root-relative resolution is an alternative not yet chosen.
