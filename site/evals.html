Evals

Reproducible, inspectable, honest measurement — not a leaderboard.

status: planned (Milestones 8–9, 12) — not implemented yet

Brief explanation

An eval runs one or more agents over a suite of tasks and scores each run. Deterministic mock and scripted baselines establish a floor and exercise the harness without any model.

Deeper technical explanation

Planned signals per run:

SignalSource
Task success (pass/fail)test command / file checks
Diff conformancegit diff vs. constraints
Tool-call count, command counttrace
Runtimewall-clock, bounded by timeout
Failure reasontrace + scorer

Regression comparison classifies tasks as improved, regressed, unchanged, or flaky across two result files, with a rerun mode for flakiness analysis.

Current implementation status

The single-run primitive exists (Milestone 4): code-agent-runtime run <task> --agent mock|scripted drives one agent over one task and returns a structured RunResult with the per-run signals the eval will aggregate — outcome (pass/fail/not-scored), tool-call count, command count, the final diff, and the test result. Minimal scoring (test_command, none) is built in. The suite runner, full scoring methods, and Markdown/HTML reports land in Milestone 8; regression/flakiness in Milestone 9; the reproducible case study in Milestone 12.

Current test/result status

Runtime tests exercise single runs of the mock and scripted agents over the shipped tasks (tests/test_runtime.py): the scripted agent passes the bugfix and CLI tasks, the mock agent fails the bugfix. No multi-agent suite report is generated yet; that validation arrives with Milestone 8.

Open questions / limitations

  • Small task counts make results illustrative, not statistically powerful.
  • A passing test command does not guarantee a correct change.
