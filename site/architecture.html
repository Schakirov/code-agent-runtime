Architecture

How a versioned task and an agent become a scored, replayable trace.

status: Milestone 4 — tasks, tools, and the runtime loop implemented; tracing/sandbox pending

Brief explanation

The runtime is an explicit pipeline. Each stage has a clear input and output so that every step can be inspected, traced, and reproduced.

Design principles: deterministic-first, everything-is-traced, separation of policy and mechanism, and no hidden cost.

Core tools (Milestone 3)

Tools are the only way an agent acts on a workspace. Every tool shares one shape — a static ToolSpec, a validated run() entry point, and a JSON-serialisable ToolResult — so the runtime can dispatch them uniformly and the tracer can record them faithfully. A ToolRegistry maps names to tools and dispatches a ToolCall. Path-taking tools are confined to one workspace directory; subprocess tools bound their output and time and run without a shell.

ToolCategoryWhat it returns
read_filereadcontent, bytes, lines, binary/truncated flags
search_reporeadmatching lines (path/line/text), counts
git_diffreadunified diff, per-file +/- counts, untracked files
write_filewritebytes written, created vs. overwrote
apply_patchwriteper-file create/modify/delete, +/- counts
run_shellexecargv, exit code, stdout/stderr, timed-out flag
run_testsexecas run_shell, plus a passed boolean

The registry's REGISTERED_TOOL_NAMES is the authoritative tool vocabulary; a test asserts it equals the task schema's KNOWN_TOOLS, so a task can never request a tool that does not exist.

Runtime loop (Milestone 4)

The runtime turns a validated task and an agent into a single structured, scored RunResult. An agent only proposes the next action (run a tool, or finish); the runtime owns every side effect — preparing the workspace, gating each call against the task's allowed_tools, dispatching it, running the task's test command, capturing the final diff, scoring, and reporting. Two deterministic, model-free agents ship: mock (a null baseline that does nothing) and scripted (replays a fixed plan, with built-in solutions for the example tasks). Run one with code-agent-runtime run <task>.

load task -> prepare workspace -> [agent step -> gate -> execute tool]* -> run tests -> score -> report
StageWhat happens
prepare workspacecopy the fixture to a fresh temp dir; git init + commit
agent stepagent.propose(Observation) returns one AgentAction
execute toolgate against allowed_tools, then dispatch via the registry
run testsrun the task's test_command through the run_tests tool
scoretest_command → pass/fail; none → not scored
reportbuild the RunResult; render text or JSON

Containment here is workspace confinement plus capability gating only — there is no process isolation, resource limit, or network policy yet (that is the sandbox, Milestone 7). Scoring is intentionally minimal; the full scorer, suite runner, and reports are Milestone 8.

Current test/result status

Scaffold-layout, task-format, tool, and runtime tests pass under python3 -m pytest -q. tests/test_runtime.py covers the agent interface, workspace preparation, allowed-tool gating, scoring, and full runs: the scripted agent fixes the tiny_python_bug fixture end to end (failing → passing) and the final diff is captured, while the mock agent leaves it failing. Tests needing git skip when it is absent.

Open questions / limitations

  • The runtime provides workspace confinement and capability gating only — no process isolation, resource limits, or network policy yet. That is Milestone 7 (sandbox).
  • Scoring covers test_command and none only; the other methods and the suite/regression machinery are Milestone 8.
  • apply_patch applies exact-context unified diffs (no fuzzy matching, no binary patches, no rename detection).
  • What is the minimal trace schema that still supports faithful replay? The RunResult shape is the first draft of that answer.
