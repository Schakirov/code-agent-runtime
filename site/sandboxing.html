Sandboxing

Blast-radius reduction and observability for semi-trusted agents.

status: planned (Milestone 7) — not implemented yet

This is not a production-grade sandbox. The goal is to reduce accidental damage and make actions observable — not to contain malicious native code. See Security.

Brief explanation

The sandbox decides what an agent is allowed to read, write, and run, and records what it actually did. Tools execute; the sandbox enforces policy.

Deeper technical explanation

Planned containment primitives:

  • Allowed read paths, allowed write paths, denied paths.
  • Command allowlist and denylist.
  • Network policy flag (default: no network).
  • Max runtime and max output size.
  • Secret scanner over inputs, outputs, and traces.
  • Optional containerized runner for stronger isolation.

Unsafe examples (reading .env, reading ~/.aws/credentials, writing outside the workspace, a dangerous shell command) will demonstrate that blocked actions are blocked while allowed actions still work.

Current implementation status

Runtime containment is not implemented. Today's guards are repo-level: .gitignore and .claude/settings.json deny rules, plus a repository hygiene scanner (Milestone 1) that flags secrets and junk in tracked files. That scanner is a commit-time guard, not a runtime sandbox — the containment primitives above are still planned.

Relevant files

  • src/code_agent_runtime/hygiene.py (commit-hygiene guard, Milestone 1)
  • src/code_agent_runtime/sandbox/ (runtime controls, planned)
  • .gitignore, .claude/settings.json
  • docs/SECURITY_MODEL.md

Current test/result status

No sandbox tests yet. Validation will assert blocked-vs-allowed behavior on unsafe demos.

Open questions / limitations

  • How much isolation is achievable without containers or root?
  • Right default network posture for scripted vs. LLM agents.
